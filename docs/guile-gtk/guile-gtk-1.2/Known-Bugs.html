14 Known Bugs

gdk-colormap-alloc-color
gdk-colormap-alloc-colors
In Gtk 1.2.10, gdk_colormap_alloc_color and gdk_colormap_alloc_colors have bugs in allocating writable colours from a shared (ie. non-private) colormap in a pseudocolor visual. The success flags and failcount returned indicate failure even when the operation succeeds.

Guile Gtk makes no attempt to do anything about this. The only suggestion would be for an application to check for a change in the pixel fields of the GdkColor objects, since this ends up being done only on success.

GdkPixmap, GdkBitmap
Pixmaps and bitmaps are not properly freed when garbage collected. A warning is printed about an undestroyed window reaching a zero reference count, and the underlying X server object is not freed.

This is due to those objects being dereferenced with gdk_window_unref instead of gdk_pixmap_unref. Although GdkPixmap and GdkWindow are the same pointer type (in Gtk 1.2), the unref functions are not the same. Guile Gtk will need to represent them as distinct classes, and allow both in functions taking GdkDrawable. (Distinct classes will also have the happy effect of stronger checking for instance in gdk-window-map which should reject a GdkPixmap.)

GtkEditable insert-text signal
The callback for the insert-text signal from a GtkEditable doesn't respect the length for the new text, and doesn't provide a means to modify the position (which is an in/out parameter).

This may change in the future, probably to a single string parameter for the text and some sort of modifiable location for the position (a one element vector perhaps).
